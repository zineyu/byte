# Implementation Plan: Core Coding Loop

**Branch**: `001-core-coding-loop` | **Date**: 2026-07-16 | **Spec**: [specs/001-core-coding-loop/spec.md](specs/001-core-coding-loop/spec.md)

**Input**: Feature specification from `/home/zine/space/project/byte/specs/001-core-coding-loop/spec.md`

## Summary

Complete the daemon-side Core Coding Loop so that a single Run can make multiple Model Turns through model output, requested tool calls, executed tool results, and continued model reasoning, until a final assistant response or cancellation. Work is confined to `byte-core` and `byte-models` tests and a new integration-style demo test; no RPC transport or frontend changes are required. The plan focuses on two known gaps: (1) correct cancellation semantics that do not persist partial assistant messages, and (2) an end-to-end read ‚Üí edit ‚Üí command demo test that exercises the full loop.

## Technical Context

**Language/Version**: Rust 2021 edition, Tauri v2 / React / TypeScript desktop shell (no frontend changes for this feature).

**Primary Dependencies**: `tokio` async runtime, `futures` streams, `serde_json` for tool arguments, `byte-protocol`, `byte-models`, `byte-tools`, `byte-skills`, `byte-session`, `byte-core`, `byte-daemon`.

**Project Type**: Rust workspace with Tauri desktop app.

**Scale/Scope**: Single-Developer, local Sessions. No multi-user, no background runs, no new provider families or transports, no hard limit on Model Turns.

## Phase 0: Research Findings

Research was limited to the current implementation and the feature specification. No external technology or unfamiliar dependency was introduced. Findings are recorded in `specs/001-core-coding-loop/research.md`.

Key decisions:
- **Decision**: On cancellation, flush buffered deltas as runtime events but do not persist the partial assistant message.
- **Rationale**: The spec explicitly forbids partial assistant messages in durable Session history (FR-012). The existing implementation persists partial content; this must be changed.
- **Decision**: The full demo test uses a composite `ModelProvider` mock (or extended `EchoProvider`) that returns a deterministic sequence: read_file ‚Üí apply_patch ‚Üí run_command ‚Üí final text.
- **Rationale**: The real OpenAI-compatible provider is not deterministic enough for a regression test; a mock provider is the project's existing testing pattern and is sufficient to verify the orchestration contract.
- **Decision**: Do not introduce a hard safety limit on Model Turns or tool-call cycles in this iteration.
- **Rationale**: The user explicitly removed this requirement. The design keeps the loop unbounded; future iterations may add provider/prompt-level controls or a soft operational limit if needed.
- **Decision**: Do not introduce new protocol types or change the persisted message format.
- **Rationale**: The existing `Message`, `MessageBlock`, `LlmMessage`, and `ToolCall`/`ToolResult` shapes already support the loop. Avoiding format changes preserves backward compatibility and respects the constitution's shared-protocol gate.

## Phase 1: Design & Contracts

### Data Model

See `specs/001-core-coding-loop/data-model.md` for entity details.

Summary:
- **Session**: durable JSONL tree with header, active path, and one optional active Run.
- **Run**: in-memory state in `SessionRunner` with `RunId` and `CancellationToken`.
- **Message**: persisted entry with `role`, `body` (blocks), `parent_id`, and optional `tool_call_id`.
- **LlmMessage**: ephemeral provider context with roles `system`, `developer`, `assistant`, `tool`; includes assistant tool calls and tool results.
- **ToolCall**: model-requested action with `id`, `name`, `arguments`.
- **ToolResult / ToolOutputResult**: completed tool outcome, with output text, error flag, and optional exit code.
- **RuntimeEvent**: ephemeral UI projection of Run lifecycle, message lifecycle, and tool lifecycle events.

State transitions for a Run:

```text
Idle ‚Üí RunStarted ‚Üí ModelTurn ‚Üí AssistantMessagePersisted ‚Üí ToolStarted ‚Üí ToolOutputDelta* ‚Üí ToolFinished ‚Üí ToolResultPersisted ‚Üí [ModelTurn ...] ‚Üí RunFinished(Succeeded)
                                                                                ‚Üì
Idle ‚Üí RunStarted ‚Üí ... ‚Üí RunCancelled ‚Üí RunFinished(Cancelled)
                                                                                ‚Üì
Idle ‚Üí RunStarted ‚Üí ... ‚Üí Error ‚Üí RunFinished(Failed)
```

The `active_run` mutex in `SessionRunner` enforces: `Idle ‚Üí Active`, and `Active ‚Üí Idle` only after a terminal outcome.

### Contracts

See `specs/001-core-coding-loop/contracts/loop.md` and `contracts/cancellation.md`.

Summary of key interfaces:

- `ModelProvider::send_message(messages, tools) -> ProviderStream`: must yield exactly one `MessageStarted`/`MessageCompleted` pair per Model Turn; `MessageCompleted` may carry `tool_calls`. The runner is responsible for continuing the loop when tool calls are present.
- `Tool::invoke(call, session_ctx, cancel_token) -> ToolOutputStream`: emits `Chunk` events and exactly one terminal `Done`. The runner maps these to `ToolStarted`, `ToolOutputDelta`, `ToolFinished`, and persists a `tool` role message with `tool_call_id`.
- `SessionRunner::send_message(params) -> RunId`: rejects concurrent runs with `RunnerError::Busy`; appends the developer message; spawns `RunExecutor`.
- `SessionRunner::cancel_run() -> CancelRunResult`: triggers the per-run cancellation token and waits for the run task to clear `active_run`.

### Quickstart Validation

See `specs/001-core-coding-loop/quickstart.md`.

Summary:
- Run `just verify` to confirm all existing tests pass.
- Run `cargo test -p byte-core runner::` to execute the new and existing loop tests.
- Run a manual demo via `just start-daemon` and the desktop UI, sending a request that requires reading a file, editing it, and running a command.
- Expect: one `RunStarted`, one or more `MessageCompleted`, `ToolStarted`/`ToolFinished` pairs, and a final `RunFinished(Succeeded)`.
- Expect cancellation: a `RunCancelled` event, no persisted partial assistant message, and a subsequent `RunFinished(Cancelled)`.
- Expect concurrency: a second `send_message` to the same Session while one Run is active returns `Busy` without changing history.
